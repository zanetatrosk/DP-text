\chapter{Testování}
\begin{chapterabstract}
Tato kapitola popisuje testování aplikace. Nejprve jsou zadefinovány pojmy týkající se testování. Poté je uvedeno, jakým způsobem bylo testování funkcionalit systému prováděno a proč byl zvolen právě tento způsob testování. Závěrečná část kapitoly se věnuje uživatelskému testování, jež bylo prováděno s potenciálními uživateli aplikace. Na konci kapitoly jsou shrnuty výsledky testování.
\end{chapterabstract}

\section{Testování}
Testování je proces, při kterém se ověřuje a verifikuje, zda software splňuje požadavky a funguje správně. Primárním přínosem testování je doručení kvalitního softwaru díky včasnému odhalení chyb a zlepšení výkonu. V dnešním době je testování běžně součástí vývoje a již nebývá finální krokem před vydáním softwaru, ale je prováděno průběžně během vývoje. \cite{ibm_software_testing}

Testy bývají rozdělovány primárně do dvou kategorií \cite{ibm_software_testing}: 
\begin{itemize}
    \item \textbf{Manuální testování} – testování, které je prováděno testerem ručně bez použití automatizovaných nástrojů. Testeři ručně simulují klikání tlačítek, vyplňování textu a tím simulují chování koncového uživatele. Z mojí zkušenosti se manuální testování používá pro finální ověření nové funkcionality, před vydáním nové verze aplikace anebo testování částí systému, kde je obtížné vytvořit automatizované testy.
    \item \textbf{Automatizované testování} – testování, které je prováděno pomocí nástrojů a skriptů, které automaticky spouštějí testy na daném softwaru. Automatizované testy bývají rychlejší a jsou vhodné zejména pro repetitivní testovací scénáře. Umožňují snížit riziko lidské chyby a konzistentně testovat systém.
\end{itemize}

\subsection{Typy testů}
V následující sekci uvedu typy testů a krátce popíšu, co každý z nich znamená. Existuje mnoho různých typů testů, ale pro účely této práce se zaměřím na ty nejběžnější, které jsou relevantní pro testování webových aplikací. 
Mezi tyto typy testů \cite{perfecto_types_of_testing} patří:
\begin{itemize}
    \item \textbf{Unit testy} – testy, které se zaměřují na testování jednotlivých částí kódu za cílem ověření, že samostatné části systému fungují správně jak mají. Díky tomu se testování zrychluje a omezuje se spouštění testů, které by jinak mohly být zbytečné.
    \item \textbf{Integration testy} – testy, které se zaobírají na testování rozhraní obsahující více částí aplikací, modelů nebo komponent za cílem ověřit, že systém funguje správně jako celek \cite{integration_testing_guide}.
    \item \textbf{End-to-end testy} – testy, které ověřují, že proces v aplikaci od začátku do konce funguje správně.
    \item \textbf{Smoke Testing} – testy, které validující stabilitu systému. Bývají spouštěny na prvotní verzi aplikace pro kontrolu, zda systém obsahuje základní funkcionality.
    \item \textbf{Acceptance Testing} – testy, které bývají prováděny stakeholdery či koncovými uživateli za účelem ověření, že systém splňuje požadavky a je připraven k nasazení.
\end{itemize}

\subsection{Způsob testování}
Pro testování funkcionalit systému jsem zvolila kombinaci manuálního a automatizovaného testování. V případě unit testů bylo zvoleno automatizované testování, protože umožňuje rychlé a efektivní testování jednotlivých částí kódu. Pro integration testy a end-to-end testy jsem zvolila manuální testování, protože tyto testy často vyžadují komplexní scénáře a interakce mezi různými částmi systému, které mohou být obtížné automatizovat. Při přidání nové funkcionality provádím vždy manuální smoke testy, abych ověřila, že základní funkcionality systému fungují správně. Před vydáním prototypu aplikace jsem se rozhodla provést akceptační testy (v textu dále označované jako uživatelské) s potenciálními uživateli. Cílem bylo získat zpětnou vazbu a ověřit, že systém splňuje stanovené požadavky a bude v budoucnu připraven k nasazení do produkčního prostředí.

\section{Implementace testů pro jednotlivé části systému}
V následujících podsekcích popíši implementaci automatizovaných unit testů pro frontend a backend. Vysvětlím, jaké nástroje jsem použila pro testování a jak jsem postupovala při psaní testů pro jednotlivé části systému.
\subsection{Frontend}
Pro frontend unit testing jsem použila testovací framework Vitest \cite{vitest_docs}, který je součástí ekosystému TanStack. Psaní testů usnadňovalo fakt, že jsem měla zkušenosti s podobným testovacím frameworkem zvaným Jest. 

Při psaní unit testů pro frontend jsem se zaměřila především na části obsahující logiku. Tyto funkce jsou většinou umístěny v adresářích \texttt{utils}. V kódu \ref{lst:frontend-unit-test} je možné vidět příklad unit testu pro funkci, která zobrazuje správný text na základě ceny akce z backendu. Test kontroluje, že pro akci s cenou 0 Kč se zobrazí text \textquote{Free}, pro akci s cenou 100 Kč se zobrazí text \textquote{100 Kč} a pro akci s nedefinovanou cenou se zobrazí text \textquote{Not specified}.

\begin{lstlisting}[language=TypeScript, caption={Příklad unit testu pro frontend}, label={lst:frontend-unit-test}]
import { describe, expect, it } from 'vitest'
import { getLabelFromPrice } from './getLabelFromPrice'

describe('getLabelFromPrice', () => {
  it('returns "Not specified" when price is undefined', () => {
    expect(getLabelFromPrice(undefined, 'USD')).toBe('Not specified')
  })

  it('returns "Not specified" when currency is missing', () => {
    expect(getLabelFromPrice(10, undefined)).toBe('Not specified')
  })

  it('returns "Free" when price is 0 and currency is present', () => {
    expect(getLabelFromPrice(0, 'CZK')).toBe('Free')
  })

  it('formats a non-zero price with two decimals', () => {
    expect(getLabelFromPrice(12.5, 'EUR')).toBe('12.50 EUR')
    expect(getLabelFromPrice(12, 'EUR')).toBe('12.00 EUR')
  })
})

\end{lstlisting}
\subsection{Backend}
Na unit testování backendu jsem vzhledem k tomu, že je backend psaný ve frameworku Spring Boot, zvolila testovací framework JUnit, se kterým jsem se setkala standardně v rámci backend vývoje ve Spring Bootu. \cite{junit_docs}

Při psaní unit testů pro backend jsem se zaměřila na testování především služeb, které obsahují \textit{business} logiku. 

\section{Uživatelské testování}
Uživatelské testování bylo prováděno s potenciálními uživateli aplikace, kteří byli vybráni na základě jejich zájmu o danou problematiku. Uživatelské testování probíhalo formou individuálních sezení, během kterých uživatelé prováděli různé úkoly v aplikaci, zatímco já jsem sledovala jejich interakce a sbírala zpětnou vazbu. Na konci těchto sezení jsem kladla otázky týkající se jejich zkušeností s aplikací, co se jim líbilo a co by mohlo být zlepšeno.

Před zahájením samotného testování byly navrženy scénáře úloh, které měli účastníci během testování plnit. Scénáře byly rozděleny do dvou skupin na scénáře pro běžného uživatele (tanečníka) a scénáře pro organizátora akcí. Snažila jsem se scénáře navrhnout tak, aby otestovaly klíčové funkce aplikace z pohledu obou typů uživatelů, a zároveň aby byly dostatečně jednoduché na pochopení a nebyly příliš časově náročné.

\subsection{Scénáře pro běžného uživatele}
Tyto scénáře byly navrženy tak, aby otestovaly různé funkce aplikace z pohledu běžného uživatele, který je tanečníkem, ale neorganizuje akce.

    \item Vyberte akci, která vás zajímá a zaklikněte, že máte o ni zájem.
    \item Přejděte na stránku s vašimi akcemi a zkontrolujte, zda akce, kterou jste zaklikli, že máte o ni zájem, je zobrazena mezi akcemi, které vás zajímají.

\subsection{Výsledky testování}
Během uživatelského testování byla získána cenná zpětná vazba od všech zúčastněných respondentů. Většina uživatelů pozitivně hodnotila jednoduchost a přehlednost uživatelského rozhraní a neměla potíže s orientací v aplikaci. V průběhu testování však byly identifikovány i dílčí nedostatky, které byly následně odstraněny.

Jedním z problémů byla vizuální nejednoznačnost karet událostí – při najetí kurzoru se zobrazovala animace stínu, která evokovala interaktivitu, ačkoliv karta samotná nebyla klikací. Tento nedostatek byl vyřešen odstraněním dané animace. Další nejasnost se týkala tlačítka pro zobrazení detailu akce, které obsahovalo ikonu šipky naznačující otevření odkazu v nové záložce, přestože se detail zobrazoval ve stejném okně. Pro zvýšení konzistence uživatelského rozhraní byla tato ikona odstraněna.

Uživatelé rovněž upozornili na nedostatek v implementaci filtrování událostí, které nebylo řešeno pomocí query parametrů. V důsledku toho docházelo při použití navigačního tlačítka „zpět“ v prohlížeči ke ztrátě nastavených filtrů. Tento problém byl odstraněn zavedením filtrování prostřednictvím query parametrů, čímž byla zajištěna perzistence uživatelského nastavení.

V neposlední řadě byl upraven mechanismus vyhledávání lokací. Původní implementace zobrazovala informaci o nenalezení výsledků i v situaci, kdy samotné vyhledávání ještě neproběhlo. K tomu docházelo z důvodu, že frontend zobrazoval tuto zprávu již při změně hodnoty ve vstupním poli, pokud byl současně počet zobrazených lokací roven nule. Tato logika byla upravena tak, aby odpovídala očekávanému chování. Informace o nenalezení výsledků se nyní zobrazuje pouze v případě, kdy uživatel zadá vyhledávací dotaz a následně proběhne vyhledávání, které skutečně nevrátí žádné výsledky.

Při testování s organizátory se ukázalo, že proces publikace události pomocí Google formuláře není zcela intuitivní. Někteří uživatelé měli potíže s pochopením, jak získat ID formuláře. Proto jsem přidala do modálu pro publikaci události pomocí Google formuláře podrobnější popis, jak ID formuláře získat.

Dále uživatelé vyjádřili výhrady k nedostatečnému zastoupení tanečních stylů. Konkrétně požadovali, aby seznam neobsahoval pouze obecné styly, jako je bachata, ale také jejich subžánry a fusion varianty (podrobněji popsáno v sekci \ref{sec:tanecni-styly}). Na základě této zpětné vazby byl seznam tanečních stylů rozšířen.

Současně byla upravena i kategorizace úrovní pokročilosti, kde byla přidána možnost „Open level“. Tato volba umožňuje organizátorům označit akce určené pro účastníky všech úrovní, aniž by bylo nutné specifikovat konkrétní pokročilost.
