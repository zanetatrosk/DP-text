\subsection{Přihlášení uživatele}
Přihlášení uživatele bylo realizováno pomocí Google na základě požadavku \ref{req:F3}. Jelikož aplikace není jen klientská, ale i serverová, bylo potřeba implementovat přihlášení i na serveru.

Prvním krokem bylo vytvoření projektu v Google Cloud Console. Poté bylo potřeba vytvořit OAuth 2.0 Client ID pro webovou aplikaci, což nám poskytlo Client ID a Client Secret, které jsou nezbytné pro implementaci přihlášení. Následně byly nastaveny redirect URI, tedy adresy, na které bude Google přesměrovávat uživatele po úspěšném přihlášení, a origins, tedy adresy, ze kterých bude možné přihlášení provést.

Poté byly přidány tzv. scopes, které určují, jaké informace o uživateli bude aplikace požadovat. V našem případě budou při prvotním přihlášení požadovány pouze základní informace o uživateli, jako je jeho jméno a emailová adresa. Tudíž se jedná o následující scopes \cite{google_oauth_scopes}:
\begin{itemize}
    \item \texttt{openid} - Tento scope propojuje uživatele s jeho osobními údaji na Google účtu.
    \item \texttt{profile} - Tento scope umožňuje získat osobní údaje uživatele, i které zveřejnil
    \item \texttt{email} - Tento scope umožňuje získat primární emailovou adresu uživatele.
\end{itemize} 

Tyto scope se řadí mezi nesenzitivní oprávnění, což jsou oprávnění, která slouží k autorizaci uživatele. 
\cite{google_drive_auth_guide}

Při nasazení aplikace do produkce a využívání Google API společnost Google u těchto oprávnění nevyžaduje ověření. Tudíž je jednodušší aplikaci uvést do produkce.
\cite{google_app_verification}

\subsubsection{Inkrementální autorizace} \label{auth_incremental}
Aplikace bude kromě zmíněných oprávněních z předešlé sekce také vyžadovat oprávnění pro přístup k Google Forms API, aby uživatelé mohli propojit formulář a odpovědi z formuláře s aplikací. Oprávnění jsou následující \cite{google_oauth_scopes}:
\begin{itemize}
    \item \texttt{https://www.googleapis.com/auth/forms.body} - Tento scope umožňuje aplikaci číst a upravovat obsah formulářů, jako jsou otázky,
    \item \texttt{https://www.googleapis.com/auth/forms.responses.readonly} - Tento scope umožňuje aplikaci číst odpovědi z formulářů, ale neumožňuje je upravovat.
\end{itemize}

Jelikož se může stát že uživatel nebude nikdy potřebovat propojit formulář s aplikací, bylo by zbytečné požadovat tato oprávnění již při prvotním přihlášení. Proto je implementována tzv. inkrementální autorizace, která umožňuje požádat o další oprávnění až v momentě, kdy jsou potřeba. V našem případě se tedy o tato oprávnění požádá až v momentě, kdy se uživatel bude snažit propojit formulář s aplikací. Pokud uživatel tato oprávnění již udělil, nebude o ně požádán znovu. \cite{google_oauth_incremental_auth}

Obecně inkrementální autorizace funguje tak, že uživatel požádá o oprávnění, která potřebuje a endpoint Google vrátí kód, který dále může být vyměněn za access token obsahující všechny oprávnění, která uživatel udělil. Tento způsob je od Google považován jako doporučený, oproti například požádání všech oprávnění najednou. \cite{google_oauth_incremental_auth} 

\subsubsection{Proces přihlášení}
Proces přihlášení začíná na straně klienta. Uživatel klikne na tlačítko Sign in a klient zavolá Google API pro přihlášení.

Url, které klient volá pro přihlášení, vygeneruje funkce na základě parametrů, jejíž implementaci lze nahlédnout v kódu \ref{lst:google_oauth_url}.

\begin{lstlisting}[language=TypeScript, label={lst:google_oauth_url}, caption={Funkce pro generování url pro přihlášení přes Google OAuth}]
export const buildGoogleOAuthUrl = (params: {
  scope: string;
  state?: string;
}) => {
  const baseUrl = GOOGLE_OAUTH_URL;
  const urlParams = new URLSearchParams({
    client_id: GOOGLE_CLIENT_ID,
    redirect_uri: GOOGLE_REDIRECT_URI,
    response_type: 'code',
    scope: params.scope,
    access_type: 'offline',
    include_granted_scopes: true.toString()
  });

  if (params.state) {
    urlParams.append('state', params.state);
  }

  return `${baseUrl}?${urlParams.toString()}`;
};
\end{lstlisting}

Funkci jsem si naprogramovala sama i přesto, že existuje například knihovna \texttt{@react-oauth/google}, která zjednodušuje proces přihlášení. Avšak při implementaci jsem narazila na problém, že tato knihovna v případě, že využíváme authorization code flow, automaticky posílá parametr prompt=consent, což způsobuje, že uživatel musí při každém přihlášení proklikat souhlas s oprávněními, i když je již udělil. Tudíž jsem se rozhodla implementovat přihlášení sama, abych měla plnou kontrolu nad tím, jaké parametry jsou posílány, a mohla tak využít inkrementální autorizaci.

Parametry, které jsou posílány v url pro přihlášení, jsou následující \cite{google_oauth_creating_client}:
\begin{itemize}
    \item \texttt{client\_id} - Toto je Client ID, které jsme získali při vytvoření OAuth 2.0 Client ID v Google Cloud Console.
    \item \texttt{redirect\_uri} - Toto je URI, na které bude Google přesměrovávat uživatele po úspěšném přihlášení. Toto URI musí být stejné jako jedno z redirect URI, které jsme nastavili v Google Cloud Console.
    \item \texttt{response\_type} - Tento parametr určuje, jaký typ odpovědi očekáváme od Google. V našem případě je to \texttt{code}, což znamená, že očekáváme autorizační kód, který následně vyměníme za access token. Druhým typem odpovědi je \texttt{token}, což znamená, že očekáváme přímo access token. Tento typ odpovědi se používá v tzv. implicitním flow, které se však nedoporučuje pro webové aplikace, protože není tak bezpečné jako authorization code flow.
    \item \texttt{scope} - Tento parametr obsahuje seznam oprávnění, o která žádáme. Oprávnění jsou oddělena mezerami.
    \item \texttt{access\_type} - Tento parametr určuje, zda chceme získat refresh token. Pokud je tento parametr nastaven na \texttt{offline}, Google nám pošle refresh token, který můžeme použít k získání nového access tokenu, když ten původní vyprší. Parametr jsem takto zvolila kvůli tomu, že bude vyžadováno, abychom v budoucnu volali API i bez přítomnosti uživatele, například pro synchronizaci odpovědí z formuláře, a v takovém případě je potřeba mít možnost získat nový access token bez toho, aby se uživatel musel znovu přihlásit.
    \item \texttt{include\_granted\_scopes} - Tento parametr určuje, zda chceme, aby Google zahrnul již udělená oprávnění do access tokenu. Pokud je tento parametr nastaven na \texttt{true}, Google nám pošle access token, který obsahuje všechna oprávnění, která uživatel udělil, nejen ta, o která právě žádáme. Při implementaci se mi stalo, že jsem na tento parametr zapomněla, a v důsledku toho uživatel při každém přihlášení resetoval všechna oprávnění, která udělil, a musel je znovu udělovat, což samozřejmě nebylo ideální.
    \item \texttt{state} - Tento parametr je volitelný, ale doporučený, a slouží k ochraně proti CSRF útokům.
Klient do něj vloží náhodný řetězec, který si uloží (např. do session).
Po přesměrování zpět ho porovná s hodnotou v odpovědi.
Tím ověří, že odpověď odpovídá požadavku iniciovanému stejným uživatelem, a ne cizímu požadavku.
\end{itemize}

Potom co uživatel zvolí účet, kterým se chce přihlásit, a udělí požadovaná oprávnění (pokud je ještě neudělil), Google přesměruje uživatele na redirect URI, které jsme nastavili, a v URL bude obsažen autorizační kód. Klient tento kód zachytí a pošle ho na náš server, kde ho vyměníme za access token. Pokud se jedná o prvotní přihlášení, Google nám pošle i refresh token, který uložíme do databáze pro pozdější použití. Poté obratem klientovi vrátíme serverem vygenerovaný vlastní JWT token, který klient bude používat pro autentizaci a autorizaci při dalších požadavcích na server. \cite{google_oauth_creating_client}

Při testování přihlášení jsem narazila na problém, že když se po nějaké době přihlásím za uživatele, který měl již odsouhlasená práva na Google Forms API, po určité době k tomuto API nedostávám přístup a Google API mi vrací token obsahující pouze základní oprávnění. Po prozkoumání jsem zjistila, že tím, že je aplikace v tzv. testovacím režimu, Google maže udělená oprávnění (kromě základních) po 7 dnech neaktivity uživatele. \cite{google_oauth_publishing_status_testing}

\subsection{Registrace uživatele}

\subsection{Integrace s Google Forms}
Jelikož jedna z hlavních funkcí aplikace je možnost propojit formulář z Google Forms (viz v požadavku \ref{req:F16}), bylo potřeba implementovat integraci s Google Forms API. Tato integrace nám umožní získávat strukturu formuláře a odpovědi, které uživatelé do formuláře zadali, a tyto informace následně zobrazovat v aplikaci.

Provolání samotného Google API je obdobné jako u autentizace, jen je potřeba, aby uživatelova oprávnění obsahovala oprávnění pro přístup k Google Forms API, což jsem řešila pomocí inkrementální autorizace, jak jsem popsala v sekci \ref{auth_incremental}, a dále je nutné znát ID formuláře. \cite{google_forms_get_method}

V rámci integrace provoláváme zejména dva endpointy Google Forms API \cite{google_forms_api_rest}:
\begin{itemize}
  \item \texttt{GET /v1/forms/\{formId\}} -- vrací formulář.
  \item \texttt{GET /v1/forms/\{formId\}/responses} -- vrací seznam odpovědí odeslaných z formuláře.
\end{itemize}

\noindent
V implementaci nejprve načítáme strukturu formuláře (\texttt{GET /v1/forms/\{formId\}}) a následně odpovědi (\texttt{GET /v1/forms/\{formId\}/responses}). Obě odpovědi jsou z Google API vráceny ve formátu JSON, který následně mapujeme do interní struktury, kterou v aplikaci používáme.

\paragraph{1) Struktura odpovědi \texttt{GET /v1/forms/\{formId\}} (zjednodušeně)}
\begin{verbatim}
{
  "formId": "1Qw8cABqYjtjOvValF28DpAEuZ2q4jwerABQ_v2jtYcc",
  "revisionId": "...",
  "info": {
    "documentTitle": "testik"
  },
  "items": [
    {
      "itemId": "0c9e2e5a",
      "questionItem": {
        "question": {
          "questionId": "id1",
          "required": true,
          "textQuestion": {}
        }
      },
      "title": "Zadejte jmeno a prijmeni"
    }
  ]
}
\end{verbatim}

\noindent
Z této odpovědi zpracováváme zejména \cite{google_forms_form_resource}:
\begin{itemize}
    \item \texttt{revisionId} \- pro detekci změny formuláře,
    \item \texttt{items[*].questionItem.question.questionId} \- interní identifikátor otázky,
    \item \texttt{items[*].title} \- text hlavičky/sloupce,
    \item typ otázky (\texttt{textQuestion}, \texttt{choiceQuestion}, \texttt{dateQuestion}, \dots) \- podle něj volíme typ interní hlavičky.
\end{itemize}

\paragraph{2) Struktura odpovědi \texttt{GET /v1/forms/\{formId\}/responses} (zjednodušeně)}
\begin{verbatim}
{
  "responses": [
    {
      "responseId": "...",
      "respondentEmail": "user@example.com",
      "lastSubmittedTime": "2026-01-10T12:34:56Z",
      "createTime": "2025-11-26T20:25:39.414Z",
      "answers": {
        "id1": {
          "questionId": "id1",
          "textAnswers": {
            "answers": [
              {
                "value": "..."
              }
            ]
          }
        }
      },
    }
  ]
}
\end{verbatim}

\noindent
Z odpovědí zpracováváme zejména \cite{google_forms_formresponse}:
\begin{itemize}
    \item \texttt{responseId} \- unikátní identifikátor odpovědi (ukládán k registraci),
    \item \texttt{respondentEmail} \- email respondenta (pokud je dostupný),
    \item \texttt{lastSubmittedTime} \- čas posledního odeslání (pro detekci aktualizace),
    \item \texttt{answers} \- mapování \texttt{questionId} $\rightarrow$ hodnota odpovědi.
\end{itemize}

Po úspěšném napojení na Google Forms API jsem nejprve naivně navrhla, že pokaždé, když organizátor navštíví stránku a zavolá GET Registrations, se zavolá Google API. Toto řešení sice zaručovalo, že má organizátor aktuální strukturu formuláře i odpovědi, avšak pokaždé bylo nutné provést HTTP požadavek a parsovat odpověď z Google API do struktury, kterou používá naše API. To způsobilo, že odpověď trvala déle (naměřila jsem 2 sekundy v prohlížeči). Dalším důvodem, proč jsem začala optimalizovat, je to, že Google má určité kvóty na volání API. Tudíž se vyplatí požadavky šetřit. Rozhodla jsem se tedy pro způsob, že strukturu formuláře i odpovědi cachuji. Vzhledem k tomu, že každá odpověď z formuláře Google odpovídá právě jednomu záznamu v databázi, budeme pro tyto účely ukládat \texttt{responseId} společně s celým obsahem dané odpovědi. Podobným způsobem si budu ukládat i strukturu formuláře v tabulce \texttt{registrations\_settings}.
Oboje je uloženo v databázi ve formátu JSON pomocí datového typu JSONB pro efektivnější načítání a parsování \cite{dudycz_postgresql_jsonb}. Po této optimalizaci se doba požadavku změnila na 300 ms. Avšak stále zůstal problém, kdy formulář synchronizovat. Rozhodla jsem se, že implementuji job, který se bude spouštět ve stanoveném intervalu (já jsem zvolila jednou za hodinu) a na pozadí synchronizovat jak formulář, tak i odpovědi z něj. Synchronizovat mohu i přesto, že je uživatel neaktivní, protože mám uložený refresh token, který mi umožňuje získat nový access token a zavolat Google API.

Pokud zjistím, že je požadavků na stanovenou kvótu příliš mnoho, mohu dobu spouštění upravit. Problém však bude nastávat v situacích, kdy organizátor potřebuje synchronizovat registrace ihned (např. pokud akce začíná a je možné, že se někdo přihlásil v časovém okně, kde ještě nedošlo k synchronizaci). Pro takové případy jsem se rozhodla implementovat endpoint, který umožní synchronizaci spustit ihned. Tato změna se projeví i v uživatelském rozhraní, kde bude implementováno tlačítko, které endpoint zavolá.

Při synchronizaci formuláře kontroluji, zda se změnila struktura formuláře, což zjišťuji porovnáním \texttt{revisionId} z Google API a \texttt{revisionId} uloženého v databázi. Pokud se \texttt{revisionId} liší, znamená to, že došlo ke změně formuláře a je potřeba aktualizovat strukturu formuláře v databázi. Pokud se \texttt{revisionId} neliší, nemusím aktualizovat strukturu formuláře, protože vím, že se nezměnila. Podobným způsobem kontroluji i odpovědi, kde porovnávám čas posledního odeslání odpovědi z Google API a čas poslední synchronizace odpovědi v databázi.

\section{Frontend}
Jak už bylo řečeno v sekci Návrh, frontend část projektu bude napsána ve frameworku React. Pro založení projektu jsem se rozhodla použít Tanstack Start, což je full-stack React framework založený na dvou klíčových technologiích: Tanstack Router pro routování a Vite pro build projektu. \cite{tanstack_react_overview}

Tanstack Start nabízí několik výhod, jako je například rychlý vývoj a flexibilní nasazení kompatibilní s Vite, jednoduché routování díky Tanstack Router, zabudované nástroje pro vývojáře a také integrace s knihovnami pro efektivní načítání dat z API (Tanstack Query, Apollo). \cite{tanstack_react_comparison}

\subsection{API na lokace}
Při tvorbě taneční akce je potřeba zadat lokaci, kde se bude konat. Jelikož nechci aby uživatelé zadávali lokaci ručně (je to nepohodlné a může to vést k chybám), bylo potřeba najít správné API, které umožňuje získávat lokace zadané uživatelem. Zároveň ale pořád chci umožnit lokaci ručně upravit, protože je možné, že některé lokace budou v API chybět (například se bude v API nacházet ulice, ale ne s číslem popisným, na které se akce bude konat). V následujících odstavcích popíšu, mezi jakými API jsem vybírala a která byla nakonec zvolena.

\subsubsection{Google places API (New)}
Jedná se o vystavenou službu od společnosti Google, která umožňuje pracovat s lokacemi. Toto API nabízí širokou škálu funkcí \cite{google_places_api_overview} jako například: 
\begin{itemize}
    \item Place Details (New) - API umožňuje získat podrobné informace o konkrétním místě, jako je adresa, telefonní číslo, hodnocení, fotografie a další.
    \item Text Search (New) - API umožňuje hledat místa na základě textového dotazu, jako je název místa, adresa nebo kategorie.
    \item Nearby Search (New) - API umožňuje hledat místa v okolí zadaných geografických souřadnic.
    \item Autocomplete (New) - API umožňuje získat návrhy míst na základě částečného textového dotazu a geografického rozsahu.
\end{itemize} 
S tímto API jsem se již setkala v několika aplikacích při zadávání lokace pro události (například v aplikaci OnlyDance).

Toto API jsem nakonec nezvolila, kvůli tomu, že je placené a cena se odvíjí od počtu požadavků, které jsou na API posílány. Jelikož nechci aby projekt se stal finančně náročným, rozhodla jsem podívat se po nějaké levnější alternativě, která by mi poskytla podobné funkce.\cite{google_maps_places_pricing}

\subsubsection{Photon}
Toto API je open-source a je založeno na OpenStreetMap datech. API nabízí funkce jako například:
\begin{itemize}
    \item Geocoding
    \item Reverse Geocoding
    \item hledání míst - podporuje i v rámci několika jazyků
\end{itemize}
Tento projekt má i vystavené zdarma API, které je možné využívat bez nutnosti hostovat vlastní instanci. Avšak je třeba dbát na to nezahlcovat toto API přílišným množstvím požadavků, protože na to server není připraven a jedná se spíše o ukázku toho, jak API funguje. Pokud by bylo třeba posílat větší množství požadavků, tak je v projektu zdokumentováno jak hostovat vlastní instanci. \cite{komoot_photon}

Nakonec jsem se rozhodla pro využití tohoto API, protože je zdarma a poskytuje všechny funkce, které potřebuji pro zadávání lokace pro události. Jelikož moje aplikace je ve fázi prototypu, nebude na API posíláno příliš mnoho požadavků, takže by neměl být problém s využíváním tohoto API. V budoucnu, pokud by se aplikace stala populární a bylo by potřeba posílat větší množství požadavků, bych zvážila hostování vlastní instance tohoto API nebo přechod na placené API, jako je Google Places API. Navíc při zadávání lokace bude nastavený přiměřený \textit{debounce}, aby se zabránilo posílání příliš mnoha požadavků na API.

\subsection{Formuláře}
Aplikace obsahuje několik formulářů, ať už se jedná o formulář pro vytvoření akce, nastavení registrace, úprava profilu a další. U jednoduchých formulářů jako je úprava profilu, stačilo stav formuláře řešit pomocí useState, ale u složitějších formulářů jako je například formulář pro vytvoření akce, jsem chtěla použít knihovnu, která by mi usnadnila práci s formulářem a zároveň by mi poskytla funkce jako například validace. 

Po prozkoumání několika knihoven jsem se rozhodla pro Tanstack Form, protože mám dobré zkušenosti s Tanstackem jako takovým a tato knihovna nabízí potřebné funkce pro práci s formuláři jako validace, error handling, správa stavu formuláře a další. Navíc je tato knihovna dobře zdokumentovaná, což usnadňuje její použití. \cite{tanstack_form}

\subsection{Načítání dat z API}
Pro načítání dat z API jsem se rozhodla použít knihovnu Tanstack Query, která poskytuje funkce pro efektivní načítání dat z API, jako je například cachování, synchronizace dat, stránkování a další. Tato knihovna mi umožňuje jednoduše načítat data z API a zároveň se stará o další aspekty načítání dat, jako je například zobrazení loading stavu, error stavu a další. \cite{tanstack_query_react_overview}

V kódu \ref{lst:profile-route} je ukázka toho, jak načítám data z API pomocí Tanstack Query v routě pro profil uživatele. 

\subsection{Routování}
Tím, že jsme vytvořili projekt pomocí TanstackStart, máme již předpřipravenou funkcionalitu pro routování v aplikaci, protože Tanstack Start zahrnuje Tanstack Router. Tato knihovna používá tzv. file-based routing, což znamená, že struktura souborů v adresáři \texttt{src/routes} určuje strukturu URL v aplikaci. \cite{tanstack_react_routing_guide}

Například soubor \texttt{src/routes/my-profile.tsx} bude odpovídat URL \texttt{/my-profile}. V ukázce kódu \ref{lst:profile-route} lze vidět definovanou routu, kde v případě že uživatel není přihlášen, bude přesměrován na přihlašovací stránku, a pokud je přihlášen, načtou se data z API a zobrazí se profilová stránka s těmito daty. 

\begin{lstlisting}[language=TypeScript, caption={Příklad definice routy pro profil uživatele}, label={lst:profile-route}]
export const Route = createFileRoute("/my-profile")({
  beforeLoad: requireAuth,
  component: RouteComponent,
});

function RouteComponent() {
  const { user } = useUser();
  const { data } = useQuery({
    queryKey: ["profile", user?.userId],
    queryFn: () => getUserById(user?.userId),
  })
  
  if (!user || !data) {
    return null;
  }
  return <ProfilePage userId={user.userId} profileDataDefault={data} key={user.userId} />;
}
\end{lstlisting}

\section{Databáze}
Jak již bylo zmíněno v sekci Návrh, pro databázi jsem zvolila relační databázi PostgreSQL. Při implementace databáze jsem vytvořila dva skripty pro vytvoření tabulek a vložení testovacích dat. První skript \texttt{create\_tables.sql} obsahuje SQL příkazy pro vytvoření všech tabulek podle schématu navrženého v sekci Návrh. Druhý skript \texttt{insert\_test\_data.sql} obsahuje SQL příkazy pro vložení testovacích dat do tabulek. 

Pro média jsem se rozhodla využít objektové úložiště MinIO, jak jsem popsala v sekci Návrh. Při ukládání médií uložím do databáze pouze URL, které odkazuje na dané médium v objektové databázi. Samotné médium je však uloženo v objektové databázi. Při implementaci formuláře pro událost jsem narazila na problém, že média ukládám rovnou na server, avšak se může stát, že uživatel nakonec událost nevytvoří. Pro tyto případy jsem se rozhodla implementovat job, který bude pravidelně kontrolovat média, která jsou uložena na serveru, a pokud zjistí, že nějaké médium není přiřazeno k žádné entitě v databázi (například události), tak toto médium smaže. Tento job se spuští jednou za den, aby se zabránilo zbytečnému zatěžování serveru. 

\section{Spuštění projektu}
Ačkoliv nasazení aplikace do produkčního prostředí není předmětem této práce, bylo třeba zajistit, aby aplikaci bylo možné snadno spustit. Pro tento účel byl zvolen Docker, který umožňuje spustit aplikaci v izolovaném kontejneru, čímž zjednodušuje proces nasazení a přispívá k zajištění konzistentního chování napříč různými prostředími \cite{docker_fast_consistent_delivery}.

Pro obě části projektu (frontend i backend) byly vytvořeny Dockerfile soubory, které slouží k definici a sestavení image dané aplikace. Dále byly vytvořeny Docker Compose soubory, jež umožňují definovat a spravovat více vzájemně propojených kontejnerů tvořících aplikaci. \cite{theserverside_dockerfile_vs_compose}
